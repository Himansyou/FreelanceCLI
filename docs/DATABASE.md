# Database Design

## 1. Database per Service

- **auth_db** (PostgreSQL): Used only by Auth Service.
- **tracking_db** (PostgreSQL): Used only by Tracking Service.
- Report Service does not own a database; it reads from Tracking Service API and caches in Redis.

---

## 2. Auth Service – `auth_db`

### Table: `users`

| Column         | Type         | Constraints | Description                    |
|----------------|--------------|-------------|--------------------------------|
| id             | UUID         | PRIMARY KEY | Generated by application       |
| email          | VARCHAR(255) | UNIQUE NOT NULL | Login email                |
| password_hash  | VARCHAR(255) | NOT NULL    | BCrypt hash                    |
| created_at     | TIMESTAMP    | NOT NULL    | Creation time                  |

**Indexes:**

- `idx_users_email` on `email` (unique, for login lookup).

---

## 3. Tracking Service – `tracking_db`

### Table: `sessions`

Stores work sessions. Designed for idempotent sync: client sends a stable `id` (e.g. UUID generated at session start); server uses ON CONFLICT (id) DO UPDATE or equivalent to make sync idempotent.

| Column            | Type         | Constraints | Description                          |
|-------------------|--------------|-------------|--------------------------------------|
| id                | UUID         | PRIMARY KEY | Client-provided or server-generated (idempotency key) |
| user_id           | UUID         | NOT NULL    | References auth user (logical FK only) |
| project_id        | VARCHAR(255) | NOT NULL    | Project name or ID                   |
| start_time        | TIMESTAMP    | NOT NULL    | Session start (UTC)                  |
| end_time          | TIMESTAMP    | NOT NULL    | Session end (UTC)                    |
| duration_minutes  | INT          | NOT NULL    | Computed: end - start in minutes     |
| device_id         | VARCHAR(255) |             | Optional device identifier           |
| created_at        | TIMESTAMP    | NOT NULL    | Server-side creation time            |

**Indexes:**

- `idx_sessions_user_id` on `user_id` (for “my sessions” and report queries).
- `idx_sessions_user_project` on `(user_id, project_id)` (for project-scoped reports).
- `idx_sessions_user_start` on `(user_id, start_time)` (for date-range reports).

We use `project_id` as VARCHAR to keep the model simple (project names or slugs). No separate `projects` table in the minimal design; can be added later.

---

## 4. Redis (Report Service)

- **Key pattern**: `report:summary:{userId}:{project}:{fromDate}:{toDate}`.
- **Value**: JSON of the report payload.
- **TTL**: e.g. 300 seconds (5 minutes); optionally invalidate on new session (e.g. delete keys by pattern or pub/sub).

---

## 5. CLI Local Storage – SQLite

The CLI uses a local SQLite file (e.g. `~/.freelancecli/sessions.db` or project-local).

### Table: `sessions`

| Column       | Type      | Description                                |
|-------------|-----------|--------------------------------------------|
| id          | TEXT (UUID) | Primary key; used for idempotent sync    |
| project     | TEXT      | Project name                               |
| start_time  | TEXT (ISO) | Session start                              |
| end_time    | TEXT (ISO) | Session end (null if running)             |
| duration_minutes | INT   | Filled when session stopped                |
| device_id   | TEXT      | Optional                                   |
| synced      | INT (0/1) | 0 = pending sync, 1 = synced               |
| created_at  | TEXT (ISO) | Local creation time                      |

### Table or file: `config` / `auth`

Store after login:

- `access_token` (JWT)
- `user_id` (optional, for convenience)
- `api_base_url` (optional)

Can be one row in a `config` table or a JSON file under `~/.freelancecli/config.json`.
